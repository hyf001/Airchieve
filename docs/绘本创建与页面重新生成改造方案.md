# 绘本创建与页面重新生成改造方案

## 1. 背景

当前绘本创建流程只生成故事正文页。封面和封底依赖编辑器中的手动入口补充：

- 手动生成封面：`POST /api/v1/storybooks/{id}/cover/generate`
- 手动生成封底：`POST /api/v1/storybooks/{id}/backcover/generate`

这套方式存在几个问题：

1. 手动创建封面链路已有 bug，不希望继续维护独立的手动封面能力。
2. 封面、正文页、封底在产品语义上都是页面，但现在编辑能力不统一。
3. 封底是固定图片，不需要生成；后续修改应通过现有图层系统叠加完成。
4. 后续希望每个页面都可以独立重新生成，包括分镜、页面文本、页面图片。

因此，本次改造目标不是继续增强“生成封面/生成封底”两个特殊功能，而是把它们收敛到统一的页面模型和页面级重新生成能力中。

## 2. 改造目标

### 2.1 创建流程目标

用户点击开始创作后，最终自动得到完整绘本：

```text
0: cover       封面
1...N: content 正文页
N+1: back_cover 封底
```

创建时：

1. 正文页按当前故事分镜生成。
2. 封面自动生成，不再需要手动入口。
3. 封底从配置中读取对应比例的 OSS 图片 URL，直接作为 `back_cover` 页面插入。
4. 创建完成后绘本状态为 `finished`，页面导航中直接包含封面、正文、封底。

### 2.2 编辑流程目标

编辑器不再提供独立的“生成封面”和“生成封底”功能。

统一为：

- 选中封面或正文页，可以使用“重新生成”能力。
- 选中封底页，通过图层工具叠加文字、图片、Logo、二维码等内容。
- 导出时继续按“页面底图 + 图层”合成。

### 2.3 页面重新生成目标

每个可 AI 生成的页面支持独立重新生成：

1. 分镜重新生成，可带用户手动调整指令。
2. 页面文本重新生成，可带用户手动调整指令。
3. 页面图片重新生成，可带用户手动调整指令。

封面也走同一套页面重新生成能力，只是后端根据 `page_type=cover` 使用封面专用策略。

封底不走 AI 重新生成。封底底图固定，修改通过图层完成。

## 3. 推荐创建流程

推荐采用“先生成正文图片，再生成封面图片”的流程，但在分镜阶段先生成封面分镜。

完整流程：

```text
1. 用户输入故事需求
2. 生成故事正文
3. 生成正文分镜，同时生成封面分镜
4. 用户确认或调整封面分镜、正文分镜
5. 创建绘本记录，同时保存封面、正文、封底全部页面占位数据
6. 后台逐页生成正文图片
7. 正文图完成后，自动选择正文参考图
8. 使用封面分镜 + 正文参考图生成封面图片
9. 封底页使用配置中的 OSS 固定封底图片 URL
10. 所有可生成页面完成后，设置绘本状态为 finished
```

创建流程中的交互建议：

1. 第 3 步生成分镜时，后端返回的数据中应包含 `cover storyboard` 和正文页 storyboards。
2. 第 4 步分镜确认阶段，把封面作为一张特殊页面展示在正文页之前。
3. 封面分镜默认可以折叠，避免给普通用户增加理解负担。
4. 用户需要精修时，可展开封面分镜，调整封面构图、角色、场景和氛围。
5. 正文页分镜仍按当前方式展示和编辑。
6. 用户点击“开始创作”时，前端需要把确认后的封面分镜和正文分镜一起提交到创建接口。
7. 绘本生成完成后，编辑器建议默认打开封面页，让用户第一眼看到完整绘本成品感。

第 4 步的时序说明：

```text
进入分镜确认页
-> 顶部展示封面分镜卡片，默认折叠
-> 下方展示正文页分镜列表
-> 用户可编辑封面分镜或正文分镜
-> 确认后进入创建参数页
-> 开始创作时提交 cover + content pages
```

注意：封面分镜不是在正文图生成后再补，而是在分镜确认阶段就存在。正文图生成完成后，只是利用正文参考图把这个封面分镜落成最终封面图片。

### 3.1 失败态要求

创建接口在同步阶段就应创建完整页序：

```text
0: cover       image_url=""
1...N: content image_url=""
N+1: back_cover image_url=配置中的 OSS 固定封底图
```

如果绘本制作中途失败：

1. 绘本状态设为 `error`，保留明确的 `error_message`。
2. 编辑器仍应展示所有页面，包括尚未生成图片的封面、正文页和封底。
3. 无图页面在页面导航和画布中显示“未生成”占位，而不是被过滤掉。
4. 用户可以选中任意失败或无图页面，通过“AI 调整当前页”重新制作。
5. 已成功生成的页面不应被失败流程回滚。
6. 图层数据不应因为重新制作基图而被清空。

这样即使后台任务在第 3 页、封面或其他阶段失败，用户也能看到完整书的结构，并逐页修复。

页面最终结构：

```text
cover:
  text: 书名或封面文案
  storyboard: 封面分镜
  image_url: AI 生成封面图
  layers: 用户叠加图层

content:
  text: 当前页故事文本
  storyboard: 正文分镜
  image_url: AI 生成正文图
  layers: 用户叠加图层

back_cover:
  text: ""
  storyboard: null
  image_url: 配置中的 OSS 固定封底图
  layers: 用户叠加图层
```

## 4. 封面生成策略

### 4.1 为什么不先生成封面图片

不推荐先生成封面图片，原因：

1. 角色一致性风险高  
   封面先生成时，正文角色形象和画风尚未确定。后续正文页容易与封面不一致。

2. 风格容易偏海报化  
   封面通常更精致、更强构图。如果正文跟随封面，正文页可能失去叙事插画感。

3. 失败成本更高  
   如果封面先生成得很好，但正文生成后不匹配，用户会明显感到不是同一本书。

4. 当前工程链路更适合正文先行  
   项目现有 `generate_page`、`previous_page_image`、正文参考图逻辑都更适合先建立正文视觉风格。

### 4.2 推荐折中方案

推荐：

```text
先生成封面分镜，后生成封面图片
```

分镜阶段生成一个 `cover storyboard`，但不立即生成封面图。

示例结构：

```json
{
  "cover": {
    "text": "书名或封面标题",
    "storyboard": {
      "scene": "概括整本故事的核心场景",
      "characters": "主角关键特征和动作",
      "shot": "封面式构图，主角突出，预留标题空间",
      "color": "与故事气质一致的主色调",
      "lighting": "温暖、明亮、有吸引力"
    }
  },
  "pages": []
}
```

生成封面图片时，输入为：

```text
书名
故事摘要
封面分镜
正文参考图
绘本 aspect_ratio / image_size / cli_type
```

这样既保留封面创意方向，又能用正文图保证角色和画风一致。

### 4.3 封面参考图选择

首版推荐沿用并增强当前“首 / 中 / 尾”策略。

规则：

1. 只从 `page_type=content` 的正文页中选择。
2. 只选择 `image_url` 非空的页面。
3. 最多选择 3 张参考图。
4. 正文页数小于等于 3 时，全选。
5. 正文页数大于 3 时，选择：

```text
第一张正文页
中间正文页
最后一张正文页
```

伪代码：

```python
def pick_cover_reference_pages(pages: list[Page]) -> list[Page]:
    content_pages = [
        p for p in pages
        if p.page_type == PageType.CONTENT and p.image_url
    ]

    if len(content_pages) <= 3:
        return content_pages

    mid = len(content_pages) // 2
    return [content_pages[0], content_pages[mid], content_pages[-1]]
```

二期可以在分镜阶段让模型标记封面候选页，例如 `cover_reference_score`，再按主角清晰度、核心道具、情绪代表性选择参考图。首版不建议复杂化。

交互策略：

1. 创建时不要求用户手动选择封面参考图，因为此时正文图还没有生成，用户无法判断哪张更适合作为参考。
2. 创建时封面参考图完全自动选择。
3. 编辑器中重新生成封面时，可以允许用户手动选择参考页。
4. 如果用户没有手动选择参考页，仍使用自动首 / 中 / 尾策略。

## 5. 封底策略

封底图片由人工事先上传 OSS。后端只读取配置，不生成、不上传、不扣费。

### 5.1 配置项

建议在 `app/core/config.py` 中增加：

```python
BACK_COVER_IMAGE_1_1: str = ""
BACK_COVER_IMAGE_4_3: str = ""
BACK_COVER_IMAGE_16_9: str = ""
```

对应环境变量：

```env
BACK_COVER_IMAGE_1_1=https://xxx.oss-cn-xxx.aliyuncs.com/back-covers/1_1.png
BACK_COVER_IMAGE_4_3=https://xxx.oss-cn-xxx.aliyuncs.com/back-covers/4_3.png
BACK_COVER_IMAGE_16_9=https://xxx.oss-cn-xxx.aliyuncs.com/back-covers/16_9.png
```

选择函数：

```python
def get_back_cover_image_url(aspect_ratio: str) -> str:
    mapping = {
        "1:1": settings.BACK_COVER_IMAGE_1_1,
        "4:3": settings.BACK_COVER_IMAGE_4_3,
        "16:9": settings.BACK_COVER_IMAGE_16_9,
    }
    url = mapping.get(aspect_ratio) or settings.BACK_COVER_IMAGE_16_9
    if not url:
        raise ValueError(f"未配置封底图片: {aspect_ratio}")
    return url
```

### 5.2 创建封底页

创建接口同步阶段创建封底页：

```python
Page(
    storybook_id=storybook_id,
    page_index=len(content_pages) + 1,
    text="",
    image_url=get_back_cover_image_url(aspect_ratio),
    page_type=PageType.BACK_COVER,
    storyboard=None,
)
```

封底页后续只通过图层系统修改。

### 5.3 封底页面重新生成

封底不提供 AI 重新生成。

封底也不提供单独的“生成封底”或“重置封底”接口。若确实需要重置封底底图，应作为后续图层/页面管理能力单独评估，不进入首版范围。

## 6. 页面重新生成设计

### 6.1 新增统一接口

建议新增页面级异步接口：

```http
POST /api/v1/pages/{page_id}/regenerate
```

请求体：

```json
{
  "regenerate_text": false,
  "text_instruction": "",
  "regenerate_storyboard": false,
  "storyboard_instruction": "",
  "regenerate_image": true,
  "image_instruction": "",
  "reference_page_ids": []
}
```

响应：

```json
{
  "storybook_id": 1,
  "page_id": 10,
  "status": "updating"
}
```

客户端继续轮询绘本状态：

```http
GET /api/v1/storybooks/{storybook_id}/status
```

终态后重新拉取绘本详情和当前页详情。

### 6.2 执行顺序

页面重新生成固定按以下顺序执行：

```text
text -> storyboard -> image
```

原因：

- 文本会影响分镜。
- 文本和分镜共同影响图片。
- 图片必须使用最终版本的文本和分镜。

### 6.3 正文页重新生成

对 `page_type=content`：

1. 如果 `regenerate_text=true`，基于整本故事上下文、当前页位置、用户指令生成新文本。
2. 如果 `regenerate_storyboard=true`，基于最终文本、整本故事上下文、用户指令生成新分镜。
3. 如果 `regenerate_image=true`，基于最终文本、最终分镜、参考图、用户图片指令生成新图。
4. 更新当前页对应字段。
5. 不清空、不修改已有图层。

### 6.4 封面页重新生成

对 `page_type=cover`：

1. `text` 可作为封面标题或封面文案，默认使用绘本标题。
2. `storyboard` 是封面分镜。
3. 图片重新生成时使用封面专用 prompt 或现有 `llm_client.generate_cover`。
4. 参考图默认从正文页中自动选择，也允许用户指定 `reference_page_ids`。
5. 更新封面页字段。
6. 不清空、不修改已有图层。

封面重新生成由统一页面重新生成入口触发，不再使用独立的“生成封面”接口。

### 6.5 封底页重新生成

对 `page_type=back_cover`：

- 不允许 AI 重新生成。
- 如果调用 regenerate 接口，应返回 400 或在前端禁用入口。
- 封底修改通过图层工具完成。

## 7. 页面状态与绘本状态联动

### 7.1 为什么需要页面级状态

改造后，绘本不再只是“整本一次性生成”。系统需要支持：

1. 创建时封面、正文页、封底先作为完整页序落库。
2. 后台逐页生成正文图和封面图。
3. 任意封面或正文页可以独立重新生成文本、分镜和图片。
4. 生成失败后，用户可以选中失败页或无图页逐页修复。

如果只依赖 `storybook.status` 和 `page.image_url` 推断，会出现几个问题：

1. 只能知道整本书是否在 `creating/updating/error`，无法知道具体哪一页正在生成或失败。
2. `image_url=""` 只能表示没有底图，不能区分“等待生成”“正在生成”“生成失败”。
3. 只重新生成文本或分镜时，`image_url` 可能不变，但页面确实经历了一次更新任务。
4. 前端页面导航、画布占位、失败重试都需要页面级状态。

因此建议在 `pages` 表增加页面级状态字段。

### 7.2 页面状态字段

在 `Page` 模型中新增：

```python
class PageStatus(str, Enum):
    PENDING = "pending"       # 已创建占位，等待生成
    GENERATING = "generating" # 当前页正在生成或重新生成
    FINISHED = "finished"     # 当前页已可用
    ERROR = "error"           # 当前页生成或重新生成失败
```

数据库字段：

```python
status: Mapped[str] = mapped_column(String(32), default="pending", nullable=False)
error_message: Mapped[str | None] = mapped_column(String(500), nullable=True)
```

响应 schema 中同步返回：

```python
class PageResponse(BaseModel):
    ...
    status: str = "finished"
    error_message: Optional[str] = None
```

历史数据迁移时，所有已有页面统一设置为：

```text
status = "finished"
error_message = null
```

原因是历史页面已经在用户侧作为已有作品展示。即使个别历史页缺少图片，也不应在迁移后突然显示为系统生成中的异常状态。历史绘本缺少封面或封底的问题，仍按“老绘本处理”策略由单独迁移或管理脚本补齐。

### 7.3 页面状态流转

创建同步阶段写入完整页序：

```text
cover:
  status=pending
  image_url=""

content:
  status=pending
  image_url=""

back_cover:
  status=finished
  image_url=固定 OSS 封底图
```

整本绘本后台生成正文页：

```text
开始生成某个 content 页
-> page.status=generating
-> 成功: page.image_url=生成图, page.status=finished, page.error_message=null
-> 失败: page.status=error, page.error_message=具体原因
```

正文全部成功后生成封面：

```text
cover.status=generating
-> 成功: cover.image_url=生成图, cover.status=finished, cover.error_message=null
-> 失败: cover.status=error, cover.error_message=具体原因
```

页面重新生成：

```text
调用 /pages/{page_id}/regenerate
-> page.status=generating
-> storybook.status=updating
-> 按 text -> storyboard -> image 执行
-> 成功: page.status=finished, page.error_message=null
-> 失败: page.status=error, page.error_message=具体原因
```

只重新生成文本或分镜时，也应使用同样的 `generating -> finished/error` 状态流转。页面状态表示“本次页面任务结果”，不是单纯表示图片是否存在。

封底页：

```text
back_cover.status=finished
```

封底不允许进入 AI regenerate 流程。

### 7.4 绘本状态和页面状态联动规则

绘本状态表示整本书的总体任务状态，页面状态表示单页任务状态。两者按下面规则联动。

创建流程：

```text
创建同步阶段:
  storybook.status=init
  cover/content.status=pending
  back_cover.status=finished

后台生成开始:
  storybook.status=creating

存在任意可生成页 status=generating:
  storybook.status=creating

所有可生成页 cover/content 均 status=finished:
  storybook.status=finished
  storybook.error_message=null

任意可生成页 status=error:
  storybook.status=error
  storybook.error_message=失败阶段 + 页面信息 + 错误摘要
```

页面重新生成流程：

```text
调用 regenerate 同步阶段:
  page.status=generating
  page.error_message=null
  storybook.status=updating

后台任务成功:
  page.status=finished
  page.error_message=null

  如果不存在其他 generating/error 页面:
    storybook.status=finished
    storybook.error_message=null
  如果仍存在其他 error 页面:
    storybook.status=error
    storybook.error_message=仍有页面生成失败

后台任务失败:
  page.status=error
  page.error_message=具体错误
  storybook.status=error
  storybook.error_message=当前页失败原因
```

终止流程：

```text
用户中止整本生成:
  storybook.status=terminated
  当前 generating 页可置为 error 或 pending
```

首版建议：中止时将当前 `generating` 的 `cover/content` 页置为 `pending`，表示用户后续仍可重新生成；已 `finished` 页面不回滚，`back_cover` 不变。

### 7.5 状态计算原则

不要在前端用 `image_url` 推断页面状态。前端应优先使用后端返回的 `page.status`。

后端 `GET /storybooks/{id}/status` 的进度统计建议改为：

```text
total_pages = cover/content/back_cover 总页数
generatable_pages = cover/content 页数
completed_pages = status=finished 的 cover/content 页数 + fixed back_cover 页数
failed_pages = status=error 的页数
generating_pages = status=generating 的页数
```

如果为了兼容现有 UI，`completed_pages` 仍保留总完成页概念，则封底创建时即计入完成页。

### 8.4 数据库迁移

新增迁移脚本：

```text
pages.status varchar(32) not null default 'pending'
pages.error_message varchar(500) null
```

历史数据迁移：

```sql
UPDATE pages
SET status = 'finished',
    error_message = NULL
WHERE status IS NULL OR status = '';
```

如果数据库不允许新增非空列时直接带默认值，可拆成三步：

1. 新增 nullable 字段。
2. 更新历史数据为 `finished`。
3. 将 `status` 改为 not null 并设置默认值。

```python
    ...

    ...
```

`regenerate_page_async` 同步阶段负责：

1. 校验页面存在。
2. 校验页面所属绘本属于当前用户。
3. 拒绝 `back_cover`。
4. 拒绝所有 regenerate flag 均为 `false` 的空任务。
5. 如需生成图片则检查积分。
6. 设置 `page.status=generating`、`page.error_message=null`。
7. 设置 `storybook.status=updating`。

`run_regenerate_page_background` 后台阶段负责：

1. 按 `text -> storyboard -> image` 顺序执行。
2. 将 `text_instruction`、`storyboard_instruction`、`image_instruction` 分别传入对应生成逻辑。
3. 将 `reference_page_ids` 用于封面或正文图片生成参考图，且只允许引用同一本绘本的 `content` 页。
4. 成功时设置 `page.status=finished`。
5. 失败时设置 `page.status=error` 和 `page.error_message`。
6. 根据页面状态汇总规则更新 `storybook.status`。

### 8.6 `app/services/storybook_service.py`

`storybook_service` 只负责整本绘本编排，不应直接堆页面 CRUD 或页面级重新生成逻辑。

创建同步阶段应复用 `page_service.create_page(...)` 创建：

1. `cover`
2. 所有 `content`
3. `back_cover`

不要在 `create_storybook_from_story_async` 中直接构造 `Page(...)` 并 `session.add(...)`。

调整 `run_create_storybook_from_story_background`：

```text
读取已创建的 cover/content/back_cover 页面
-> 逐页设置 content.status=generating
-> 生成正文图片并增量写入 content.image_url
-> 成功页设置 content.status=finished
-> 正文图完成后设置 cover.status=generating
-> 生成封面图并写入 cover.image_url
-> 成功后设置 cover.status=finished
-> 根据页面状态汇总设置 storybook.status
-> 扣费
```

注意：

- 生成正文图片时只处理 `content` 页。
- 生成封面时只参考已完成的 `content` 页。
- 封底页在创建同步阶段已经写入，后台任务不生成封底、不上传 OSS，状态直接为 `finished`。
- 如果正文或封面失败，将失败页面置为 `error`，绘本置为 `error`，保留所有页面和已生成图片，错误信息明确说明失败阶段和页码。
- 失败后用户可通过页面级 regenerate 对无图页面逐页重新制作。

### 9.2 新增当前页重新生成入口

2. 右侧工具面板；
3. 当前页上下文菜单。

首版推荐放在右侧工具面板，和 AI 改图、文字、绘画工具并列。

### 9.2.1 简单模式

默认展示简单模式。

核心控件：

```text

- 只改图片
- 优化文字
- 重做这一页
```

对应后端参数：

这样普通用户只需要描述目标，不需要理解文本、分镜、图片三层生成链路。

### 9.2.2 高级设置

简单模式下提供“高级设置”折叠区。

展开后展示：

### 9.2.3 按页面类型变化

```text
cover:
```

### 9.4 轮询和刷新

页面重新生成触发后：

1. 后端将当前页 `status` 设为 `generating`，将当前绘本 `status` 设为 `updating`。
2. 前端立即更新当前页本地状态，导航缩略图和画布显示生成中。
3. 开始轮询 `GET /storybooks/{id}/status`。
4. 轮询期间根据 `completed_pages`、`failed_pages`、`generating_pages` 更新全局进度。
5. 终态后重新拉取绘本详情。
6. 重新拉取当前页详情，包括图层。

重新生成图片只更新基图 `image_url`，不能清空图层。

### 9.5 页面状态 UI

前端需要在页面导航、画布、工具面板中消费 `page.status`：

```text
pending:
  页面导航显示“待生成”标记
  画布显示未生成占位
  允许触发 AI 调整当前页

generating:
  页面导航显示生成中 spinner
  画布显示生成中占位
  禁用重复 regenerate

finished:
  正常展示底图和图层

error:
  页面导航显示失败标记
  画布显示错误占位和错误摘要
  允许用户重新生成当前页
```

前端不应再用 `image_url === ""` 作为唯一状态判断。`image_url` 只能辅助判断是否有底图，页面主状态以 `page.status` 为准。

### 9.6 类型和 service 改动

`frontend/src/services/storybookService.ts` 中页面类型需要增加：

```ts
export type PageStatus = 'pending' | 'generating' | 'finished' | 'error';

export interface StorybookPage {
  ...
  status: PageStatus;
  error_message?: string | null;
}
```

涉及组件：

- `PageNavigator`
- `EditorCanvas` 或当前画布容器
- `ToolPanel` / `RegenerateTool`
- `useStorybookLoader`
- `DownloadDialog`

导出时建议跳过或阻止导出未完成页面：

2. 用户可选择取消导出并先修复页面。
3. 首版不建议导出缺底图页面，避免产物不完整。

## 10. 积分策略

建议：

1. 创建绘本按“正文页数量 + 封面页”扣费。
2. 封面制作消耗积分，和正文图片生成一样计入创建成本。
3. 固定封底不扣费。
4. 页面重新生成：
   - 只重新生成图片：按 1 次图片生成/编辑扣费。
   - 封面图片重新生成：按 1 次图片生成/编辑扣费。
   - 重新生成文本或分镜：可不扣费，或后续定义低成本扣费。
   - 文本 + 分镜 + 图片一起重新生成：按图片生成扣 1 次。

创建前必须检查 `正文页数 + 1` 的积分，避免正文生成完成后封面积分不足。

## 14. 最终产品形态

用户创建绘本时：

```text
输入故事需求
-> 确认故事和分镜
-> 可选展开并调整封面分镜
-> 开始创作
-> 自动得到：封面 + 正文页 + 固定封底
-> 默认打开封面页
```

用户编辑绘本时：

```text
选中封面:
  通过 AI 调整当前页修改封面分镜和封面图
  可在高级设置中选择正文参考页

选中正文页:
  通过 AI 调整当前页优化文字、分镜、图片

选中封底:
  通过图层工具叠加和编辑内容
```

用户导出绘本时：

```text
每一页按底图 image_url + visible layers 合成
封面、正文、封底统一参与导出
```

这套方案将封面和封底从特殊手动功能收敛为统一页面能力，减少维护分支，也为后续页面级精修提供更清晰的产品和技术边界。
