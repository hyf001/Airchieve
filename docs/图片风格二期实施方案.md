# 图片风格二期实施方案：接入绘本生成链路

## 1. 阶段目标

第二期在第一期图片风格基础能力之上，将画风接入 Web 绘本生成链路。

本阶段完成后，系统应具备：

- 生成分镜时使用画风摘要作为弱参考
- 创建绘本时选择图片风格
- 绘本创建时锁定当前画风版本
- 页面图片生成时使用锁定版本的参考图、风格描述、生成提示词、负面提示词
- 页面重新生成时沿用绘本锁定的画风版本
- Web 创建流程从 `template_id` 切换为 `image_style_id`
- 绘本生成后不支持更换画风

本阶段不做：

- 小程序改造
- 自动解析画风
- 管理员试生成
- 风格市场
- 用户自定义画风

## 2. 前置条件

第二期开始前，第一期必须已完成：

1. `ImageStyle` 表存在。
2. `ImageStyleVersion` 表存在。
3. `Storybook` 已有 `image_style_id` 和 `image_style_version_id` 字段。
4. 已有可用画风列表接口。
5. 至少存在一个 `is_active=true` 且有已发布版本的画风。
6. 参考图列表保存在 `ImageStyleVersion.reference_images` 中。

## 4. 后端 Schema 改造

### 4.1 GenerateStoryboardRequest

文件：

```text
app/schemas/storybook.py
```

新增字段：

```text
image_style_id: int
```

说明：

- 第二期新流程要求分镜生成前必须选择画风。
- 如果前端暂时存在兼容路径，可短期设为 Optional，但正式逻辑应要求必传。

### 4.2 CreateStorybookFromStoryRequest

新增字段：

```text
image_style_id: int
```

废弃字段：

```text
template_id
```

处理建议：

- 新链路中 `image_style_id` 必传。
- 如果请求仍传 `template_id`，返回 `400`。
- 错误文案：`当前生成流程已改为使用画风，请选择画风`

### 4.3 StorybookResponse

新增：

```text
image_style_id
image_style_version_id
```

可选增加：

```text
image_style_name
image_style_cover_image
```

用于前端展示当前绘本画风。

## 5. 后端 API 改造

### 5.1 生成分镜接口

现有接口：

```http
POST /api/v1/storybooks/storyboard
```

请求新增：

```json
{
  "image_style_id": 1
}
```

改造文件：

```text
app/api/v1/storybook_api.py
```

处理逻辑：

1. 接收 `image_style_id`。
2. 调用 `create_storyboard_only()` 时传入 `image_style_id`。
3. 如果画风不存在、停用或没有发布版本，返回明确错误。

### 5.2 基于故事创建绘本接口

现有接口：

```http
POST /api/v1/storybooks/from-story
```

请求新增：

```json
{
  "image_style_id": 1
}
```

改造文件：

```text
app/api/v1/storybook_api.py
```

处理逻辑：

1. 接收 `image_style_id`。
2. 不再查询 `Template`。
3. 调用 `create_storybook_from_story_async()` 时传入 `image_style_id`。
4. 后台任务传入 `image_style_version_id`，或在后台根据绘本 ID 读取锁定版本。

### 5.3 页面重新生成接口

如果现有页面重新生成支持传入模板或风格参数，需要禁止。

规则：

- 页面重新生成不允许传新的 `image_style_id`。
- 页面重新生成不允许切换到当前最新版本。
- 只能使用绘本已锁定的 `image_style_version_id`。

错误文案：

```text
绘本生成后不支持更换画风
```

## 6. Service 改造

### 6.1 create_storyboard_only

现有：

```text
create_storyboard_only(story_content, page_count, cli_type)
```

改为：

```text
create_storyboard_only(story_content, page_count, cli_type, image_style_id)
```

处理步骤：

1. 调用 `image_style_service.validate_style_available(image_style_id)`。
2. 取当前版本的 `style_summary`。
3. 根据画风分类或版本配置生成 `storyboard_complexity` 建议。
4. 调用 LLM 客户端生成分镜时传入轻量画风信息。

分镜阶段只使用：

```text
画风名称
画风摘要
分镜复杂度建议
```

分镜阶段不使用：

```text
参考图
完整风格描述
生成提示词
负面提示词
```

### 6.2 create_storybook_from_story_async

现有：

```text
create_storybook_from_story_async(..., template_id)
```

改为：

```text
create_storybook_from_story_async(..., image_style_id)
```

处理步骤：

1. 创建前校验画风可用。
2. 获取当前已发布版本。
3. 创建 `Storybook` 时写入：

```text
image_style_id
image_style_version_id
```

4. 不再写入 `template_id`。
5. 返回值不再返回 `template`，可返回：

```text
storybook_id
title
image_style_version_id
```

### 6.3 run_create_storybook_from_story_background

现有：

```text
run_create_storybook_from_story_background(..., template, images)
```

改为：

```text
run_create_storybook_from_story_background(..., image_style_version_id, images)
```

也可以只传 `storybook_id`，后台从 `Storybook.image_style_version_id` 读取。

推荐方式：

- 后台根据 `storybook_id` 查询绘本。
- 从绘本读取锁定的 `image_style_version_id`。
- 再读取完整版本配置。

这样可以避免任务参数和数据库状态不一致。

### 6.4 页面生成循环

在生成正文页时，现有调用类似：

```text
llm_client.generate_page(..., template=template, reference_images=images)
```

改为：

```text
llm_client.generate_page(
  ...,
  character_reference_images=images,
  previous_page_image=previous_page_image,
  image_style_version=image_style_version
)
```

说明：

- `images` 是用户上传的角色参考图，不是画风参考图。
- 画风参考图来自 `image_style_version.reference_images`。
- 上一页图片继续用于绘本内部连续性。

## 7. LLM Client 改造

### 7.1 llm_cli.py

文件：

```text
app/services/llm_cli.py
```

接口调整：

```text
create_storyboard_from_story(..., style_summary=None, storyboard_complexity=None)
generate_page(..., character_reference_images=None, image_style_version=None)
generate_cover(..., image_style_version=None)
```

如果需要兼容 `doubao_cli.py`，两个实现类都要同步方法签名。

### 7.2 gemini_cli.py

分镜生成：

- 在 prompt 中加入画风摘要。
- 明确画风只影响画面表达复杂度，不改变故事内容。
- 极简、低龄、涂鸦类画风弱化复杂镜头和光影。

页面图片生成：

- system instruction 中加入风格描述。
- prompt 中加入生成提示词。
- prompt 中加入负面提示词。
- parts 中加入画风参考图。
- 画风参考图前增加说明：只学习整体视觉风格，不复制具体人物、文字、构图或场景。

参考图分组建议：

```text
1. 用户上传角色参考图
2. 上一页图片
3. 画风参考图
```

不要把三类参考图混成一个含义。

### 7.3 doubao_cli.py

如果当前豆包链路也用于绘本生成，需要同步：

- 方法签名
- 分镜画风摘要
- 图片生成画风参考图
- 异常处理

如果豆包暂不支持完整链路，需要在接口层限制或返回明确错误。

## 9. 异常处理

### 9.1 未选择画风

前端：

- 禁用继续按钮，提示 `请选择画风`。

后端：

- 返回 `422`。

### 9.2 画风被停用

场景：

- 用户打开页面时画风可用
- 创建绘本时管理员已停用

处理：

- 后端返回 `400`
- 前端提示 `该画风已停用，请重新选择画风`
- 前端重新拉取画风列表

### 9.3 画风版本发生变化

场景：

- 用户生成分镜时使用 v1
- 创建绘本时管理员发布了 v2

推荐规则：

- 创建绘本时使用当时当前版本。
- 分镜阶段只使用画风摘要，轻微版本变化可接受。
- 如果需要更严谨，分镜接口也返回 `image_style_version_id`，创建绘本时传回并锁定该版本。

二期推荐简化：

- 创建绘本时锁定当前版本。

### 9.4 画风参考图全部不可用

处理：

- 后端中止当前页面生成。
- 页面状态置为 `error`。
- 错误文案：`画风参考图不可用，请联系管理员检查画风配置`

不做纯文本替代，因为当前模型均具备多模态理解能力，图片生成阶段应使用参考图。

### 9.5 老请求仍传 template_id

处理：

- 返回 `400`
- 文案：`当前生成流程已改为使用画风，请选择画风`

### 9.6 页面重新生成传入新画风

处理：

- 返回 `400`
- 文案：`绘本生成后不支持更换画风`

## 10. 验收标准

第二期完成后，应满足：

1. Web 创建绘本必须选择画风。
2. 分镜生成接口接收 `image_style_id`。
3. 分镜 prompt 使用画风摘要。
4. 创建绘本接口接收 `image_style_id`。
5. 创建绘本时写入 `image_style_id` 和 `image_style_version_id`。
6. 页面生图使用锁定版本的参考图、风格描述、生成提示词、负面提示词。
7. 页面重新生成沿用锁定版本。
8. 绘本生成后不能更换画风。
9. 新链路不再写入 `template_id`。
10. 小程序不受本期影响。

## 11. 建议测试

后端：

- 分镜生成传入有效画风成功
- 分镜生成传入停用画风失败
- 创建绘本传入有效画风成功
- 创建绘本未传画风失败
- 创建绘本后保存锁定版本 ID
- 页面生图读取锁定版本
- 页面重新生成不允许换画风
- 画风参考图全部不可用时页面生成失败

前端：

- 创建流程加载画风列表
- 未选择画风不能创建
- 生成分镜时传 `image_style_id`
- 创建绘本时传 `image_style_id`
- 不再传 `template_id`
- 绘本详情展示当前画风

回归：

- 纯故事生成仍可用
- 分镜编辑仍可用
- 绘本生成状态轮询仍可用
- 页面错误状态展示仍可用

## 12. 不做事项

第二期明确不做：

- 小程序改造
- 自动解析画风
- 管理员试生成
- 批量重新生成
- 绘本生成后更换画风
- 风格市场
- 用户自定义画风
