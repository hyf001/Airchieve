# 图层系统改造方案

> 将编辑工具（文字、绘画、贴图等）作为独立图层持久化到后端，支持实时渲染和二次编辑。

---

## 一、改造目标

1. **图层持久化** — 文字、绘画、插入图片等操作作为独立图层存储到数据库，不再合并到原图
2. **二次编辑** — 用户可随时修改、删除、排序已有图层
3. **页面隔离** — 编辑单个页面不影响其他页面数据
4. **基图保护** — AI 生成的原图永不覆盖，图层独立于基图
5. **撤销/重做** — 前端本地操作历史，毫秒级响应
6. **导出合并** — 下载时将所有可见图层合成到一张图片

---

## 二、现状分析

### 2.1 当前数据模型

```
Storybook 表
  └── pages: JSON 数组  ←  所有页面以 JSON 存在一个字段里
        └── StorybookPage
              ├── text: str
              ├── image_url: str      ←  基图，图层合并后覆盖此字段
              ├── page_type: str
              └── storyboard: dict
```

**问题：**
- 页面是 JSON 数组，编辑单页需要重写整个 JSON
- 图层（文字、绘画）保存时合并到 image_url，无法二次编辑
- 基图被覆盖，无法回退
- AI 改图会丢失已有的文字/绘画

### 2.2 当前编辑保存流程

```
用户编辑文字/绘画 → 前端合成到 Canvas → 导出为 base64 → PUT /pages/{index} 覆盖 image_url
```

### 2.3 当前相关 API

| 端点 | 说明 |
|------|------|
| `PUT /storybooks/{id}/pages/{index}` | 保存页面（text + image_url） |
| `POST /storybooks/image/edit` | AI 改图（返回 base64，不存库） |
| `DELETE /storybooks/{id}/pages/{index}` | 删除页面 |
| `PATCH /storybooks/{id}/pages/reorder` | 页面重排序 |

## 三、目标数据模型

### 3.1 整体关系

```
Storybook (绘本)  ──1:N──>  Page (页面)  ──1:N──>  Layer (图层)
```

### 3.2 Page 表（新建，从 Storybook.pages JSON 中独立）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int PK | 页面 ID |
| storybook_id | int FK → storybooks.id | 所属绘本 |
| page_index | int | 页面顺序（从 0 开始） |
| image_url | text | AI 生成的原图（永不覆盖） |
| text | text | 故事文字 |
| page_type | varchar(32) | 页面类型：cover / content / back_cover，默认 content |
| storyboard | json (nullable) | 分镜信息 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时�� |

**索引：** `(storybook_id, page_index)` 联合唯一索引，保证同一绘本内页序不重复。

### 3.3 Layer 表（新建）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int PK | 图层 ID |
| page_id | int FK → pages.id | 所属页面 |
| layer_type | varchar(32) | 图层类型 |
| layer_index | int | 渲染顺序（越小越靠底） |
| visible | bool | 是否可见（默认 true） |
| locked | bool | 是否锁定（默认 false） |
| content | json | 图层数据（结构随 type 不同） |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

**索引：** `(page_id, layer_index)` 联合索引，保证查询排序效率。

### 3.4 Layer 类型定义

```
layer_type 取值：
  - "text"        文字图层
  - "draw"        绘画图层
  - "image"       插入图片图层
  - "sticker"     贴纸图层（预留）
  - "adjustment"  调整图层（亮度/对比度等，预留）
```

## 四、API 设计

### 4.1 设计原则

- **以 page_id 为主键**，不使用 storybook_id + page_index 组合
- **所有接口只操作目标记录**，不触碰无关数据
- **兼容现有接口**，旧接口在新接口稳定后下线

### 4.2 新增接口

#### 页面维度

```
GET    /api/v1/pages/{page_id}                  获取单个页面详情（含图层列表）
PUT    /api/v1/pages/{page_id}                  更新页面文字
POST   /api/v1/pages/{page_id}/base-image       替换基图（AI 改图后调用）
POST   /api/v1/pages/{page_id}/export            导出合并图片（服务端合成）
```

#### 图层维度

```
GET    /api/v1/pages/{page_id}/layers               获取页面所有图层（按 layer_index 排序）
POST   /api/v1/pages/{page_id}/layers               新增图层
PATCH  /api/v1/pages/{page_id}/layers/{layer_id}    更新图层（位置、内容、显隐等）
DELETE /api/v1/pages/{page_id}/layers/{layer_id}    删除图层
PATCH  /api/v1/pages/{page_id}/layers/reorder        批量调整图层顺序
```

#### 获取页面列表（按绘本维度）

```
GET    /api/v1/storybooks/{id}/pages            获取绘本的所有页面列表（按 page_index 排序）
```

### 4.3 各接口详细说明

#### GET /pages/{page_id}

**响应：**

```json
{
  "id": 1,
  "storybook_id": 10,
  "page_index": 0,
  "image_url": "https://oss.../page0.png",
  "text": "从前有座山，山里有座庙...",
  "page_type": "content",
  "storyboard": { ... },
  "layers": [
    {
      "id": 1,
      "layer_type": "text",
      "layer_index": 0,
      "visible": true,
      "locked": false,
      "content": { "x": 100, "y": 200, "text": "从前有座山", ... }
    },
    {
      "id": 2,
      "layer_type": "draw",
      "layer_index": 1,
      "visible": true,
      "locked": false,
      "content": { "strokes": [...] }
    }
  ]
}
```

#### POST /pages/{page_id}/layers

**请求体：**

```json
{
  "layer_type": "text",
  "layer_index": 0,
  "content": {
    "x": 100,
    "y": 200,
    "text": "从前有座山",
    "fontFamily": "ZCOOL KuaiLe",
    "fontSize": 24,
    "fontColor": "#333333"
  }
}
```

**响应：** 返回创建的完整 Layer 对象（含 id）。

#### PATCH /pages/{page_id}/layers/{layer_id}

**请求体：** 只传需要更新的字段。

```json
{
  "content": { "x": 150, "y": 250 },
  "visible": true
}
```

**注意：** `content` 字段的更新策略为**整体替换**，前端传完整 content（而非 merge）。

#### POST /pages/{page_id}/base-image

**请求体：**

```json
{
  "image_url": "https://oss.../new_page0.png"
}
```

**说明：** 只更新 image_url，不触碰任何图层。AI 改图后调用此接口。

#### POST /pages/{page_id}/export

**请求体：**

```json
{
  "format": "png"
}
```

**响应：** 返回合并后的图片 URL（服务端用 Canvas/Sharp 合成基图 + 可见图层）。

### 4.4 旧接口兼容策略

| 旧接口 | 处理方式 |
|--------|----------|
| `PUT /storybooks/{id}/pages/{index}` | 内部转换为 page_id 操作，过渡期保留 |
| `GET /storybooks/{id}` 中的 pages 字段 | 继续返回页面列表，但补充 layer 信息 |
| `POST /storybooks/image/edit` | 保持不变，只改返回值处理逻辑 |

## 五、前端改造

### 5.1 实时渲染架构

```
渲染层级（从底到顶）：

┌──────────────────────────────┐
│  ① base_image (img 标签)      │  ← Page.image_url
├──────────────────────────────┤
│  ② adjustment layers          │  ← CSS filter 或 Canvas
├──────────────────────────────┤
│  ③ draw layers               │  ← Canvas 叠加或 SVG
├──────────────────────────────┤
│  ④ image/sticker layers       │  ← 绝对定位 img 标签
├──────────────────────────────┤
│  ⑤ text layers               │  ← 绝对定位 div（可编辑）
└──────────────────────────────┘
```

**实现方式：**

```
<div class="canvas-container" style="position: relative">
  <img src={image_url} />                          ← 基图

  {layers
    .filter(l => l.visible)
    .sort((a, b) => a.layer_index - b.layer_index)
    .map(layer => <LayerRenderer key={layer.id} layer={layer} />  ← 各图层
  )}
</div>
```

每个图层作为独立 DOM 元素，支持：
- 绝对定位（x, y）
- 旋转（rotation）
- 缩放（width, height）
- 交互（拖拽、选中、编辑）

### 5.2 编辑工具改造

| 工具 | 当前行为 | 改造后行为 |
|------|----------|------------|
| 文字工具 | 创建文字 → 合成到图片 | 创建 text layer → 存 Layer 表 |
| 绘画工具 | 画笔 → 合成到图片 | 创建 draw layer → 存 Layer 表 |
| AI 改图 | 替换整个图片 | 只替换 image_url，图层不动 |
| 插入图片 | 无 | 创建 image layer → 存 Layer 表 |

### 5.3 图层面板

在编辑器侧边栏新增「图层面板」组件：

```
┌─ 图层面板 ─────────────┐
│                        │
│  👁 📌 文字图层 1       │  ← 可拖拽排序
│  👁 📌 绘画图层 1       │  ← 可切换显隐
│  👁 📌 插入图片 1       │  ← 可锁定
│                        │
│  ─── 基图（不可删除）── │
│  🖼 page0.png          │
│                        │
│  [+ 添加图���]          │
└────────────────────────┘
```

功能：
- 图层列表，按 layer_index 排序
- 拖拽排序 → 调用 `PATCH /layers/reorder`
- 眼睛图标 → 切换 visible → 调用 `PATCH /layers/{id}`
- 锁图标 → 切换 locked（锁定后不可编辑）
- 点击选中 → 高亮该图层，进入编辑模式
- 删除按钮 → 删除图层
- 基图信息展示（不可删除、不可排序）

### 5.4 自动保存策略

```
用户操作（拖拽/编辑/新增/删除）
  │
  ├─ 立即更新前端状态 + 重新渲染（0ms）
  │
  ├─ 记录 Command 到 Undo Stack（0ms）
  │
  └─ 防抖 1.5s → 调用后端 API 保存
                   │
                   ├─ 成功 → 无操作
                   └─ 失败 → 显示错误提示，重试
```

**注意：** 撤销/重做操作不额外防抖，直接取消之前的防抖任务，用新状态发起保存。

### 5.5 撤销/重做实现

#### Command 结构

```
Command {
  type: "add" | "delete" | "update" | "reorder" | "replaceBaseImage"
  layerId: string | null
  before: LayerSnapshot | PageSnapshot | null    // 操作前快照
  after:  LayerSnapshot | PageSnapshot | null    // 操作后快照
}
```

#### Undo Stack + Redo Stack

```
┌──────────┐         ┌──────────┐
│  Undo    │  ←──   │  Redo    │
│  Stack   │         │  Stack   │
│  (max 50)│         │  (动态)  │
└──────────┘         └──────────┘
```

#### 各操作类型的 undo/redo

| 操作 | Undo 动作 | Redo 动作 |
|------|-----------|-----------|
| 新增图层 | 移除该图层 | 重新插入（恢复 snapshot） |
| 删除图层 | 恢复图层（用 before snapshot） | 再次移除 |
| 更新图层 | 恢复 before 属性 | 应用 after 属性 |
| 排序图层 | 恢复旧顺序 | 应用新顺序 |
| 替换基图 | 恢复旧 image_url | 应用新 image_url |

#### 核心流程

```
用户操作
  ├─ 1. 捕获 before snapshot
  ├─ 2. 执行操作 → 更新前端状态 + 渲染
  ├─ 3. 生成 Command（before + after）→ 入 Undo Stack
  ├─ 4. 清空 Redo Stack
  └─ 5. 防抖保存到后端

Ctrl+Z 撤销
  ├─ 1. Undo Stack 弹出 Command
  ├─ 2. 用 Command.before 恢复前端状态 + 渲染
  ├─ 3. Command 入 Redo Stack
  └─ 4. 防抖保存到后端

Ctrl+Shift+Z 重做
  ├─ 1. Redo Stack 弹出 Command
  ├─ 2. 用 Command.after 应用到前端状态 + 渲染
  ├─ 3. Command 入 Undo Stack
  └─ 4. 防抖保存到后端
```

#### 边界情况

| 场景 | 处理方式 |
|------|----------|
| 切换页面 | 每个页面维护独立的 undo/redo 栈，切回来时恢复 |
| 关闭编辑器再打开 | 栈清空，从后端加载最终状态 |
| 栈溢出（>50步） | 丢弃栈底最早的 Command |
| 快速连续操作 | 每次 before 在操作前捕获，不丢失中间状态 |
| 防抖期间的撤销 | 取消待执行的保存，立即用新状态保存 |

### 6.2 Page 模型要点

```python
class Page(Base):
    __tablename__ = "pages"

    storybook_id: Mapped[int]    # 外键 → storybooks.id
    page_index: Mapped[int]      # 页面顺序
```

### 6.3 Layer 模型要点

```python
class Layer(Base):
    __tablename__ = "layers"

    id: Mapped[int]              # 主键
    page_id: Mapped[int]         # 外键 → pages.id
    layer_type: Mapped[str]      # text / draw / image / sticker / adjustment
    layer_index: Mapped[int]     # 渲染顺序

    locked: Mapped[bool]         # 默认 False
    content: Mapped[dict]        # JSON，结构随 layer_type 不同

```

### 6.4 数据库初始化

在 `main.py` 的 `lifespan()` 中增加 Page 和 Layer 表的自动建表逻辑，与现有 Storybook 表一致。

### 6.5 导出服务

`POST /pages/{page_id}/export` 的实现：

1. 下载 image_url 对应图片
2. 按 layer_index 顺序，依次将可见图层合成到图片上
3. 文字图层 → 用 Pillow/Sharp 绘制文字
4. 绘画图层 → 用 Pillow/Sharp 绘制线条
5. 图片图层 → 用 Pillow/Sharp 贴图
6. 输出为指定格式（png/jpg）
7. 上传到 OSS，返回 URL

## 七、数据迁移

### 7.1 迁移步骤

```
阶段 1：建表（不影响现有功能）
  ├─ 创建 pages 表
  ├─ 创建 layers 表
  └─ 现有接口继续使用 Storybook.pages JSON

阶段 2：数据迁移
  ├─ 遍历所有 storybook
  ├─ 将 Storybook.pages JSON 拆解插入 pages 表
  └─ 验证数据完整性

阶段 3：双写（过渡期）
  ├─ 旧接口写入时同步写入 pages 表
  ├─ 新接口可直接读写 pages 表
  └─ 前端逐步切换到新接口

阶段 4：切换
  ├─ 前端全部切换到新接口
  ├─ 旧接口标记 deprecated
  └─ 观察一周无问题

阶段 5：清理
  ├─ 下线旧接口
  └─ 可选：删除 Storybook.pages 字段
```

### 7.2 迁移脚本逻辑

```
对每个 Storybook：
  for i, page_data in enumerate(storybook.pages):
    Page.create(
      storybook_id = storybook.id,
      page_index = i,
      image_url = page_data.image_url,
      text = page_data.text,
      page_type = page_data.page_type,
      storyboard = page_data.storyboard,
    )
```

**注意：** 迁移时所有页面的 layers 表为空（历史图层已合并到图片中，无法恢复）。

## 十一、关键风险与应对

| 风险 | 应对 |
|------|------|
| 迁移数据丢失 | 迁移前备份 Storybook.pages JSON，保留旧字段 |
| 图层过多导致渲染卡顿 | 限制单页最大图层数（建议 20），lazy render |
| 自动保存与撤销冲突 | 撤销时取消待执行的防抖保存，立即保存新状态 |
| 导出合成效果与前端不一致 | 使用与前端相同的渲染参数（字体、颜色等） |
| 小程序端兼容 | 小程序暂不改造，继续使用旧接口，后续单独适配 |
